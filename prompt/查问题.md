
  诊断阶段（先发现问题）

  1. /deep-dive — 两阶段管线：先 trace 追踪调用链，再深度分析。最适合发现"空壳函数"和 stub 实现。
  2. /oh-my-claudecode:verify — 验证某个功能是否真正端到端工作，不只看代码存在与否。
  3. code-reviewer agent — 让专门的 reviewer agent 审查代码质量，会识别出 TODO 伪装成实现、只写了 happy
  path 等问题。

  修复阶段

  4. /deslop (ai-slop-cleaner) — 清理 AI 生成的"看起来对但实际不对"的代码模式。
  5. /ralplan — 如果问题很多，用共识规划来制定系统性的修复计划。

  推荐做法

  我建议分两步走：

  第一步：先跑一轮诊断，搞清楚问题的范围和严重程度。可以用：
  - deep-dive 分析核心模块（MCP 工具、知识飞轮、Hook 系统）
  - 找出哪些是 stub、哪些是只覆盖 happy path 的

  第二步：根据诊断结果，用 ralplan 制定修复计划，再逐步执行。

  要不要我现在就启动一轮深度审计？可以从最核心的几个模块开始（MCP 工具、飞轮流程、Hook 系统），用
  parallel agents 同时扫描，给你一个完整的问题清单。
